### Q6. « Comment architecturer un système multi-agents pour une banque suisse contrainte LPD / FINMA ? »

Walk the layers — be visual:

1. **Inférence souveraine**: LLM on-prem (Mistral, Llama 3 hébergé) ou cloud souverain (Infomaniak, Exoscale) selon classification des données. Cloud public uniquement pour les données publiques.
2. **MCP gateway** unique en sortie : tout appel agent → outils internes (core banking, CRM, data warehouse) passe par ce gateway. Authentification par identité agent, RBAC, log d'audit immuable.
3. **RAG privacy-aware** : embeddings calculés *on-prem* sur le corpus interne, retrieval avec filtres ACL au niveau document. Pas d'embedding qui sort du périmètre.
4. **Orchestration** : un agent superviseur déterministe (state graph, LangGraph-style) qui distribue les sous-tâches à des agents spécialisés. Pas d'agents qui s'auto-instancient — gouvernance.
5. **Observabilité + FinOps** : logs structurés par trace agent, coûts par requête, alerting sur dérives. C'est non négociable en banque.
6. **Human-in-the-loop** sur tout ce qui touche à la décision client ou à la donnée sensible — pas par confort, par exigence FINMA.

> *"Et concrètement, j'ai vécu chacune de ces couches : MCP gateway à Scrypt, RAG en démo chez Wonderful, audit FINMA à UBP, DevSecOps + cloud à FlowBank."*

### Q10. « FinOps IA — leviers concrets ? »

Six levers, in priority order:

1. **Routing par intention**: détection rapide (modèle small) → routage vers Haiku/Mistral-small pour 80 % du trafic, Opus seulement quand la tâche le justifie. Économie typique 60–80 %.
2. **Prompt caching** (Anthropic, OpenAI le supportent natif) : sur prompts longs récurrents, réduction des coûts d'input de 90 %. Activation triviale, ROI immédiat.
3. **Batch APIs** pour tout ce qui n'est pas temps réel (extraction nocturne, scoring, enrichissement) : 50 % de réduction.
4. **Semantic caching** sur les requêtes utilisateur similaires (vector hash → réutilisation de la réponse). Très efficace sur les assistants conversationnels internes.
5. **Right-sizing du contexte** : couper les prompts gras, n'envoyer que ce qui est utilisé. Effet immédiat sur tokens d'input.
6. **Fine-tuning vs prompting** : le fine-tuning n'est rentable que sur volume + spécialisation forte. À évaluer cas par cas, pas par défaut.

> *"Sur la Token Factory, c'est un dashboard FinOps unifié qui rend ces leviers actionnables — sans observabilité, on optimise à l'aveugle."*

### Q11. « Legacy Modernization augmentée par l'IA — votre approche ? »

Five-step pipeline, all human-in-the-loop:

1. **Cartographie augmentée** : un agent parcourt le code legacy, génère un graphe de dépendances + une doc fonctionnelle inverse (souvent inexistante). 2–3 semaines au lieu de 3 mois.
2. **Extraction de spécification** : à partir du code, l'agent reconstitue les règles métier en langage naturel — validation par le BA / métier. C'est là que la valeur émerge : la doc qui n'existait pas existe maintenant.
3. **Génération de test-harness** : avant de toucher au code, un agent génère une couverture de tests caractéristiques (golden master) — sécurité du refactor.
4. **Strangler-fig avec refactor agentique** : nouvelles features et modules réécrits en parallèle, l'ancien recule progressivement. L'agent Claude Code écrit le nouveau code, le dev review.
5. **Sortie progressive** : on ne décommissionne que quand les tests caractéristiques passent à 100 % sur le nouveau code.

> *"Le piège classique du legacy : croire que l'IA va tout réécrire toute seule. Faux. L'IA accélère 3–5x les phases de cartographie et d'extraction de spec — qui sont les plus douloureuses humainement. Le refactor lui-même reste piloté par les ingénieurs."*

### Q13. « Comment gérer l'anxiété d'une équipe face à l'automatisation ? »

Three things, learned from FlowBank (DevSecOps shift) and Scrypt (Claude Code adoption):

1. **Transparence radicale** sur l'intention : l'IA *augmente*, pas *remplace*. Si elle remplace, le dire. L'ambiguïté est ce qui crée la peur.
2. **Workshops hands-on, pas de keynotes** : on montre, on fait, on rit des erreurs de l'agent ensemble. La démystification vient de l'usage personnel.
3. **Redesign du rôle, pas du headcount** : un dev qui code 30 % moins de boilerplate fait *plus* de design, plus de revue, plus de domain. On articule explicitement la nouvelle valeur du rôle.

> *"La résistance que j'ai rencontrée n'était jamais à la techno — toujours à l'incertitude sur le rôle. Quand on adresse ça frontalement, l'adoption suit."*

### Q14. « Comment formez-vous un Business Analyst ou un PO au context engineering ? »

> "Trois shifts mentaux. Un — le BA apprend que sa connaissance métier *est* du contexte exploitable : il l'écrit dans des fichiers de mémoire (markdown structuré) lisibles par les agents, comme avant il écrivait des specs Confluence. Deux — le PO apprend à formuler des specs *exécutables par un agent* : conditions de succès claires, exemples concrets, boundaries explicites — c'est du prompt engineering appliqué. Trois — les deux apprennent à *lire* la sortie d'un agent : qualité, hallucination, cas limites. C'est une posture de reviewer, pas d'utilisateur passif. Concrètement, je le fais en 2 demi-journées : une de théorie + démo, une de pratique sur un cas réel du client."
